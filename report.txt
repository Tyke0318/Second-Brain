# 研究项目报告：基于大模型与向量数据库的全局记忆智能问答系统

## 一、 需求分析
在传统的大语言模型（LLM）对话交互中，模型缺乏长期的、个性化的状态记忆，且无法直接读取和精准定位用户的大型本地私有文档。基于此，本项目旨在构建一个**赛博记忆助手（Second Brain）**，核心需求包括：
1. **多模态知识录入**：支持碎片化短文本的快捷录入，以及 TXT、PDF 等长文档的深度解析与切片。
2. **高精度知识找回**：克服纯向量检索（Dense Retrieval）在专有名词、特定数字提取上的“语义鸿沟”问题。
3. **全局记忆细粒度管理**：用户需能够像管理数据库一样，对已存入的记忆片段进行检索、修改、溯源和批量删除。
4. **沉浸式问答交互**：基于检索增强生成（RAG）架构，结合用户的历史记忆与当前问题，提供流式问答体验。

## 二、 技术选型与整体架构
项目采用“轻量级前端 + 本地化向量引擎 + 云端大模型”的架构：
* **前端交互层**：`Streamlit`。负责构建响应式侧边栏、对话流界面以及全局记忆管理弹窗（Modal Dialog）。
* **大模型服务**：`ZhipuAI (GLM-4)`。负责最终的上下文整合与流式文本生成。
* **向量数据库**：`ChromaDB (PersistentClient)`。作为本地记忆的持久化载体，存储文本块、向量（`embedding-2`）及元数据。
* **文档解析引擎**：`PyMuPDF (fitz)`。用于高保真 PDF 文本提取。
* **分词与词频检索库**：`jieba` + `rank_bm25`。用于构建本地倒排索引，弥补向量检索的不足。

## 三、 核心模块设计与代码解析

### 1. 知识库构建与预处理模块
该模块主要解决长文档输入时的排版错乱和语义割裂问题（位于 `brain.py` 和 `app.py`）。
* **PDF 智能解析**：在 `extract_text_from_file` 函数中，弃用了粗暴的纯文本提取，改用 `fitz.open` 的 `blocks` 模式。通过核心代码 `blocks.sort(key=lambda b: (b[1], b[0]))` 按照 Y 轴和 X 轴坐标对文本块进行排序，有效解决了学术论文中常见的**双栏排版解析乱码**问题。
* **语义防断裂切片 (Chunking)**：在 `get_text_chunks` 函数中，放弃了死板的定长切片。首先按段落（`\n\n`）拆分，当单个段落超出 `chunk_size=400` 时，利用正则表达式 `re.split(r'(?<=[。！？.!?])', para)` 严格按照标点符号（句子边界）进行二次切分，并辅以 `chunk_overlap=50` 的重叠度，确保任何记忆块都具备完整的上下文语义。
* **元数据注入 (Metadata Injection)**：在 `app.py` 的文档处理循环中，通过 `enhanced_chunk = f"[Document Theme/Source: {uploaded_file.name}]\n{chunk}"` 将文件名硬编码到被向量化的文本片段前。这赋予了碎片化 Chunk 全局的“主题光环”，极大提升了后续按文档主题提问时的召回率。

### 2. RRF 混合检索引擎模块
这是本项目的核心算法突破（位于 `brain.py` 的 `query_memory` 函数）。纯 ChromaDB 向量检索对字面精准匹配能力弱，本项目设计了**双路召回 + 倒数排名融合（RRF）**机制：
* **路数一（向量检索）**：调用 ZhipuAI `embedding-2` 模型获取 Query 向量，从 ChromaDB 召回 Top-20 相似度 ID。
* **路数二（关键词检索）**：获取 ChromaDB 全量文档，利用 `jieba` 对 Query 和全局语料进行中文分词，通过 `BM25Okapi` 算法计算词频得分，召回得分大于 0 的 Top-20 ID。
* **RRF 融合排序**：设定平滑常数 `k=60`，对双路召回的结果遍历打分 `rrf_scores[doc_id] += 1.0 / (k + rank + 1)`。最终取综合得分最高的 Top-10 喂给大模型。这一设计让系统既具备“语义泛化”能力，又具备“精准死抠字眼（如专有名词、提取码）”的能力。

### 3. 全局记忆管理模块 (CRUD UI)
基于 Streamlit 的高级特性封装了独立的管理大屏（位于 `app.py` 的 `memory_management_dialog`）：
* **状态隔离与弹窗化**：通过 `@st.dialog("Global Memory Base Management", width="large")` 装饰器，将管理界面与主聊天界面物理隔离，提升操作空间。
* **多维条件检索**：既支持按照原始文件（Source）进行折叠展示和一键清空（`delete_memory_by_source`），又支持利用 ChromaDB 的 `where_document={"$contains": query_text}` 进行精确的字面量过滤搜索。
* **即时热更新**：在搜索结果中，将只读的文本替换为 `st.text_area`，并配套保存按钮。用户修改文本后触发 `update_memory`，系统会后台重新调用 Embedding 接口生成新向量并覆盖原 Chunk，实现了知识库的热修复。

### 4. 智能问答交互模块
（位于 `app.py` 聊天界面部分）。
每次用户提问时，触发检索引擎，将找回的记忆片段按照 `[时间] 来源: 内容` 格式组装为 `context_text`。
在 System Prompt 中设定严格的指令：“`Please answer the user's question based on the provided [Memory Snippets] below...`”。结合 `st.session_state.messages` 维护的短期对话历史，调用 `client.chat.completions.create(..., stream=True)` 实现基于本地记忆库流式作答，并保留拒绝幻觉的兜底逻辑。

## 四、 项目主要创新点
1. **内存级轻量混合检索 (In-Memory Hybrid Search)**：在没有部署 ElasticSearch 等重型全文搜索引擎的情况下，利用 Python 内存计算 BM25 结合 ChromaDB，以极低的架构复杂度实现了工业级的双路召回与 RRF 融合。
2. **高阶 PDF 布局还原能力**：针对 RAG 系统最头疼的解析预处理环节，通过基于坐标轴维度的 Block 排序算法，实现了高可用性的排版还原。
3. **白盒化的记忆生命周期管理**：摒弃了黑盒化的 Agent 自动删除方案，提供了透明、细粒度（精确到 Chunk 级别）的可视化前端数据修改/删除交互，保障了私有数据的绝对安全与精准控制。

## 五、 总结与展望
本项目以极简的代码结构，完整打通了从长文档解析、混合向量检索到前端交互与数据库管理的 RAG 全链路。系统不仅能作为外挂知识库运行，其代码中自带的“元数据注入”和“按句切片”等策略，完美规避了传统 RAG 开发中的常见工程陷阱。

**未来演进方向：**
1. **多模态记忆扩展**：可引入图像解析库（如 OCR）或大模型多模态能力，使 Second Brain 能够记忆图片、图表等复杂对象。
2. **动态 Chunking**：后续可探索基于文档结构的动态分块策略（例如按照 Markdown 的 Heading 层级进行切片），进一步提升检索的上下文完整性。
3. **知识图谱融合 (Graph RAG)**：提取记忆中的实体关联，从单一的向量相似度检索升级为包含逻辑推理的图谱检索。
